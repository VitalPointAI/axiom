```python
"""Duplicate transaction detector for Axiom verification pipeline.

Final safety-net scan extending Phase 2 DedupHandler. Three scan types:
  1. Within-table exact tx_hash duplicates (auto-merge, score=1.0)
  2. Cross-chain bridge heuristic (flag only, score=0.60)
  3. Exchange-vs-on-chain full re-scan (multi-signal scoring)

Balance-aware auto-merge: merges only when removing the duplicate brings
the calculated balance closer to on-chain. All decisions logged in
verification_results for audit trail.
"""
```

**Class: DuplicateDetector(pool)**

Constructor takes psycopg2 pool. Import ASSET_DECIMALS from indexers.dedup_handler for reuse.

**Public method: scan_user(user_id: int) -> dict**
  - Run all three scan types in order:
    1. _scan_hash_duplicates(user_id)
    2. _scan_bridge_duplicates(user_id)
    3. _scan_exchange_duplicates(user_id)
  - Return stats: {hash_dupes_merged, bridge_flagged, exchange_flagged, exchange_merged, total_scanned}

**Private: _scan_hash_duplicates(user_id) -> int**
  Within-table exact tx_hash duplicates. These are definite duplicates from fetcher bugs.
  ```sql
  SELECT tx_hash, chain, wallet_id, COUNT(*) as cnt, array_agg(id ORDER BY id) as ids
  FROM transactions
  WHERE user_id = %s AND tx_hash IS NOT NULL
  GROUP BY tx_hash, chain, wallet_id
  HAVING COUNT(*) > 1
  ```
  For each group: keep the row with the LOWEST id (earliest inserted).
  Soft-delete others by setting `needs_review = TRUE` and `notes = 'DUPLICATE: exact hash match with tx id={kept_id}. Score=1.0. Auto-merged.'`
  Also mark the duplicate's transaction_classifications (if any) with needs_review=True.
  Log merge in verification_results with diagnosis_category='duplicate_merged', diagnosis_detail containing both tx IDs, confidence=1.0.
  Return count of merged duplicates.

**Private: _scan_bridge_duplicates(user_id) -> int**
  Cross-chain bridge heuristic. A bridge transaction appears on source chain (direction='out') and destination chain (direction='in') with similar amount and within 30 minutes.

  Algorithm:
  1. Get all user's outgoing transfers across all chains:
     ```sql
     SELECT t.id, t.tx_hash, t.chain, t.amount, t.token_id, t.block_timestamp, t.wallet_id
     FROM transactions t
     WHERE t.wallet_id IN (SELECT id FROM wallets WHERE user_id = %s)
       AND t.direction = 'out'
       AND t.action_type IN ('TRANSFER', 'transfer')
     ORDER BY t.block_timestamp
     ```
  2. For each outgoing: search for matching incoming on a DIFFERENT chain with:
     - Same or equivalent asset (use resolve_token_symbol to normalize)
     - Amount within 1% tolerance (reuse DedupHandler._amounts_match logic inline)
     - Timestamp within 30 minutes (1800 seconds) -- bridges are slower than same-chain
  3. Score = 0.60 (medium confidence -- never auto-merge bridges)
  4. Flag both transactions with needs_review=True, notes describing potential bridge pair.
  5. Log in verification_results with diagnosis_category='duplicate_merged', diagnosis_detail with both tx IDs and chains, confidence=0.60, status='open' (needs specialist review).
  Return count of bridge pairs flagged.

**Private: _scan_exchange_duplicates(user_id) -> int**
  Full re-scan of exchange_transactions vs on-chain transactions. Extends DedupHandler but:
  - Re-scans ALL exchange txs (not just unflagged ones) as a final sweep
  - Uses multi-signal scoring instead of binary match/no-match
  - Balance-aware auto-merge for high-confidence matches

  For each exchange tx:
  1. Check Signal 4: exchange amount ~ on-chain amount (1%) + timestamp within 10 min
     If match: score = 0.80
  2. Also check Signal 2: same amount + timestamp within 10 min + same asset (between any two on-chain txs that could be dupes)
     If match: score = 0.85
  3. Check Signal 3: same amount + same day + same asset
     If match: score = 0.60

  Take the highest score from all signals.

  Apply threshold actions:
  - score >= 1.0: auto-merge (won't happen for exchange dupes, only hash dupes)
  - 0.75 <= score < 1.0: call _balance_aware_merge(). If merge improves balance, auto-merge. Else flag.
  - 0.50 <= score < 0.75: flag for specialist review (needs_review=True on exchange tx)
  - < 0.50: log only (debug level)

  Return count of flagged + merged.

**Private: _balance_aware_merge(user_id, exchange_tx_id, onchain_tx_id, chain) -> bool**
  Determines if removing the exchange duplicate improves reconciliation accuracy.
  1. Get current calculated balance (sum in - out - fees from transactions for this wallet)
  2. Get the exchange tx amount
  3. Compute balance_after_merge = calculated_balance - exchange_tx_amount (if direction was 'in') or + amount (if 'out')
  4. Get on-chain balance from the most recent verification_results for this wallet (if available)
  5. If abs(balance_after_merge - onchain) < abs(calculated_balance - onchain): merge improves accuracy -> return True
  6. If no on-chain balance available: return False (don't auto-merge without ground truth)

  If True: mark exchange tx needs_review=True with note 'Auto-merged: balance-aware merge improved accuracy by X NEAR. Score=Y.'

**Private: _log_duplicate(conn, user_id, wallet_id, chain, token_symbol, detail, confidence) -> None**
  Insert into verification_results with:
  - diagnosis_category = 'duplicate_merged'
  - diagnosis_detail = detail (JSONB with tx IDs, scores, merge decision)
  - diagnosis_confidence = confidence
  - status = 'open' if flagged, 'resolved' if auto-merged
  - verified_at = NOW()

**Important implementation notes:**
- Use Decimal for all amount comparisons (not float)
- Import ASSET_DECIMALS from indexers.dedup_handler -- single source of truth
- For exchange tx amount conversion: exchange quantities are already in human units (no decimals conversion needed). On-chain amounts need decimals conversion.
- The 30-minute window for bridges (vs 10 minutes for same-chain) accounts for cross-chain confirmation times.
- When soft-deleting duplicate transactions, do NOT actually DELETE rows. Set needs_review=True and add explanatory notes. The specialist decides final action.
- All conn usage: getconn/putconn pattern with try/finally. Commit after each scan type.
