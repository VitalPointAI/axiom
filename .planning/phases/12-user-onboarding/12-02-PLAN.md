   - 'use client' component with `useState` for current step (1-5)
   - On mount, call `Promise.all([apiClient.get('/api/wallets'), apiClient.get('/api/jobs/active'), apiClient.get('/api/preferences')])` to determine starting step

     - If has wallets + no active jobs + pipeline ran (has transactions or verification results) -> Step 5 (Review)
     - If has wallets + no active jobs + no transactions -> Step 3 (Import, they added wallets but nothing ran yet) OR Step 2 if they need to continue adding
   - Render the appropriate step component based on `step` state

   - `handleSkipToDashboard`: call `POST /api/preferences/complete-onboarding` then `router.replace('/dashboard')`
   - Show step indicator (1/5, 2/5, etc.) at top with step names

   - Added wallets shown as cards: chain badge + address (truncated) + remove button (X icon)

     - ETH/Polygon/Cronos/Optimism: format "0x followed by 40 hex chars", where to find "MetaMask -> Account details -> Copy address", what's pulled "Token transfers, DeFi interactions", example "0x742d35Cc..."
   - "Continue" button: POST /api/wallets for each pendingWallet (sequentially to avoid race conditions), then onNext(). Disable button while submitting. Show error toast if any wallet fails.
   - "Add another" is implicit (can keep adding before hitting Continue)
   - "Skip to dashboard" link -> onSkip()
   - Must have at least 1 wallet to Continue (disable Continue if empty)

If non-empty, show "We found wallets that might be yours" section with cards: address (truncated), chain badge, context ("47 transfers from vitalpointai.near"), "Add" button (POST /api/wallets), "Not Mine" button (hides card via local state, no DB write per CONTEXT.md).

   - Set expectations: "Some items may be flagged for review. This is normal — Axiom flags anything it's not 100% sure about so you can confirm."
   - "Go to Dashboard" button: POST /api/preferences/complete-onboarding then router.replace('/dashboard'). Disable after first click (isLoading state).
