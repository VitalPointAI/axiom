  <name>Task 1: StreamingWorker — asyncio long-running process for multi-chain streaming</name>
  <files>indexers/streaming_worker.py, tests/test_streaming_worker.py</files>
  <behavior>
    - StreamingWorker.start() launches asyncio tasks for each enabled streaming chain
    - StreamingWorker.stop() gracefully cancels all streaming tasks
    - On new NEAR transactions found, upserts to DB and queues classify_transactions job
    - On new EVM block, queues incremental sync job for affected wallets
    - Worker loads tracked wallets from DB (wallets table) and refreshes periodically (every 60s)
    - Worker reads chain_sync_config to determine which chains to stream
    - pg_notify('new_transactions', payload) called after successful transaction upsert
  </behavior>
  <action>
Create `indexers/streaming_worker.py`:

1. Class `StreamingWorker`:
   - __init__(self, pool, cost_tracker=None) — takes DB pool and optional cost tracker
   - `async start()` — load chain_sync_config (only enabled chains with streaming support). For each NEAR chain, create asyncio task running NearStreamFetcher.stream_blocks(). For each EVM chain with WebSocket URL, create asyncio task running EVMStreamFetcher.watch_blocks(). Store tasks in self._tasks list.
   - `async stop()` — cancel all tasks, wait for cleanup with asyncio.gather(*tasks, return_exceptions=True)
   - `async _refresh_wallets()` — every 60 seconds, re-query wallets table to get current set of tracked wallet addresses grouped by chain. Update the tracked_wallets set that streaming tasks use.
   - `async _on_near_txs_found(txs, user_id, wallet_id)` — upsert transactions to DB (ON CONFLICT DO NOTHING), call pg_notify('new_transactions', json payload with wallet_id), queue classify_transactions job if new rows inserted.
   - `async _on_evm_new_block(block_header, chain)` — for each wallet on this chain, queue an incremental_sync job (or evm_incremental) with startblock = block_header.number.

2. Entry point:
   - `async def run_streaming_worker(pool, cost_tracker=None)` — creates StreamingWorker, calls start(), waits for KeyboardInterrupt/SIGTERM, calls stop().
   - Can be launched from service.py as a separate thread/process, or as a standalone `python -m indexers.streaming_worker` command.

3. Integration with service.py:
   - Add a `--streaming` flag to service.py's argument parser. When set, launch the StreamingWorker in a background thread alongside the job queue polling loop. This allows one process to handle both job queue processing and real-time streaming.
   - Import and initialize CostTracker in service.py, pass to all fetcher constructors.
   - Load chain_sync_config at service startup and register fetchers from it (instead of hardcoded handler dict). Fall back to hardcoded dict if chain_sync_config table doesn't exist yet (migration not applied).

Write tests:
- Test start/stop lifecycle (mock fetchers, verify tasks created and cancelled)
- Test _refresh_wallets loads wallets from DB mock
- Test _on_near_txs_found calls pg_notify and queues job
- Test _on_evm_new_block queues incremental sync for correct wallets
