<objective>
Modify the Rust indexer to resolve account strings to integer IDs via the PostgreSQL dictionary table, calculate segment starts (block_height / 1000 * 1000), and emit integer pairs to stdout. Update the Python COPY pipeline (account_indexer.py and run_account_indexer.sh) to use v2 staging tables with INTEGER columns and insert into account_block_index_v2.

Purpose: This is the core indexing pipeline change. The Rust binary becomes the sole writer to account_dictionary (via INSERT ON CONFLICT RETURNING id), and all downstream COPY/INSERT targets switch to the v2 table. Dictionary resolution happens in the writer thread (single PG connection, no thread-safety complexity) with an in-memory HashMap cache for speed.

Output: Modified Rust binary, updated Python orchestrator, updated shell script.
</objective>

**main.rs changes:**

1. **Add a new CLI argument `--database-url`** (with env fallback `DATABASE_URL`). This is the PostgreSQL connection string for the dictionary table. Example: `postgresql://neartax:pass@127.0.0.1:5433/neartax`.

2. **Create a `DictionaryCache` struct** with:
   - `cache: HashMap<String, i32>` — maps account_id string to integer ID
   - `client: postgres::Client` — blocking PG connection
   - `new(database_url: &str) -> Self` — connects to PG, pre-allocates HashMap with capacity 20_000_000
   - `warm_cache(&mut self)` — loads entire `account_dictionary` table into the HashMap at startup. Run `SELECT id, account_id FROM account_dictionary` and insert each row into the HashMap. Log the number of entries loaded to stderr.
   - `resolve(&mut self, account_id: &str) -> i32` — the core method:
     1. Check `self.cache.get(account_id)` — return if found (fast path, will be >99% of calls after warm)
     2. If cache miss: `SELECT id FROM account_dictionary WHERE account_id = $1`
     3. If found in DB: insert into cache and return
     4. If not found: `INSERT INTO account_dictionary (account_id) VALUES ($1) ON CONFLICT (account_id) DO UPDATE SET account_id = EXCLUDED.account_id RETURNING id` — insert into cache and return
   - Add a simple reconnect-on-error wrapper: if any PG query fails with a connection error, attempt `Client::connect()` once, then retry. This handles idle timeouts during long HTTP fetches. Log reconnections to stderr.

3. **Modify the writer thread** to own a DictionaryCache instance:
   - Before entering the receive loop, create `DictionaryCache::new(database_url)` and call `warm_cache()`
   - The channel type remains `Vec<(String, u64)>` from workers (workers still extract string accounts + block heights — no PG access in workers)
   - In the writer loop, for each `(account_string, block_height)` pair:
     a. `let account_int = dict_cache.resolve(&account_string);`
     b. `let segment_start = (block_height / 1000) * 1000;` (integer division, then multiply)
     c. Write `"{account_int}\t{segment_start}\n"` to stdout (cast segment_start as i64 -> i32 is safe since block heights are < 200M)
   - The writer thread now outputs integer pairs instead of string pairs

4. **Update the `main()` function:**
   - Parse the new `--database-url` argument
   - Pass `database_url` to the writer thread via move closure
   - Log to stderr: "Dictionary pre-warm complete: {N} entries" after cache warm
   - Workers remain unchanged — they still extract `(String, u64)` pairs from archives

5. **Keep the existing architecture intact:**
   - Workers still send `Vec<(String, u64)>` through the channel
   - All HTTP fetching, JSON parsing, archive extraction logic stays the same
   - Progress logging, retry logic, archive URL routing — all unchanged
   - Only the writer thread's output format changes

IMPORTANT: Use `postgres::NoTls` for the connection (not TLS). The database is on localhost in production. Set `tcp_keepalive` on the connection config to handle long idle periods:
```rust
let mut config = database_url.parse::<postgres::Config>().expect("Invalid DATABASE_URL");
config.keepalives(true);
config.keepalives_idle(std::time::Duration::from_secs(30));
let client = config.connect(postgres::NoTls).expect("Failed to connect to PostgreSQL");
```

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-15-03 | Tampering | Dictionary INSERT from Rust | mitigate | Account IDs come from NEAR blockchain archives (trusted source). Parameterized queries prevent SQL injection. ON CONFLICT handles race conditions. |
| T-15-04 | Denial of Service | Dictionary table lock during bulk INSERT | mitigate | Use ON CONFLICT DO UPDATE (not DO NOTHING) to avoid serialization failures. Pre-warm cache minimizes inserts during bulk indexing. |
| T-15-05 | Information Disclosure | DATABASE_URL in CLI args | mitigate | Use --database-url with env var fallback (DATABASE_URL). Process listing shows the arg, but this is a single-user server. Same pattern as existing FASTNEAR_API_KEY handling. |
| T-15-06 | Denial of Service | PG connection idle timeout | mitigate | Set tcp_keepalive=true, keepalives_idle=30s on postgres::Config. Reconnect-on-error wrapper retries once on connection failure. |
</threat_model>
