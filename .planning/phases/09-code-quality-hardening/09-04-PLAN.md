   def mock_other_user():
       """Second user context for cross-user isolation tests."""

5. **Create `tests/test_api_authorization.py`** with tests for each protected endpoint:

   The strategy: User 1 (mock_user, user_id=1) "owns" data. User 999 (mock_other_user) tries to access it. The DB mock should be configured so queries filtering by user_id=999 return empty results (simulating proper WHERE user_id = %s filtering).

   Test cases:
   - `test_user_cannot_see_other_users_wallets` — GET /api/wallets as user 999 returns empty list (not user 1's wallets)
   - `test_user_cannot_access_other_users_wallet_by_id` — GET /api/wallets/{wallet_id_owned_by_user_1} as user 999 returns 404 (wallet not found for this user)
   - `test_user_cannot_see_other_users_transactions` — GET /api/transactions as user 999 returns empty list
   - `test_user_cannot_modify_other_users_transaction` — PATCH /api/transactions/{tx_id} as user 999 returns 404
   - `test_user_cannot_generate_other_users_reports` — POST /api/reports/generate as user 999 operates on their own data only
   - `test_user_cannot_see_other_users_verification` — GET /api/verification/status as user 999 returns their own (empty) status

   For each test:
   - Create an `auth_client_other` by overriding `get_current_user` and `get_effective_user` with `mock_other_user`
   - Configure `mock_cursor.fetchall.return_value = []` and `mock_cursor.fetchone.return_value = None` to simulate user 999 having no data
   - Assert the response is 200 with empty data, or 404 — NOT 200 with user 1's data

   The key assertion: verify that the SQL query executed by the router includes the user_id parameter matching the authenticated user. Use `mock_cursor.execute.call_args` to inspect the SQL and parameters.
